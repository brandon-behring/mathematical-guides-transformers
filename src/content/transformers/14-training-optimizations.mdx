**Where we are.** Scaling laws allocate a training-compute budget between parameters and tokens; this chapter asks whether that run fits on available devices. We begin with one memory equation separating parameters, gradients, optimizer state, and activations. Four techniques then modify distinct terms. Mixed precision changes storage width while preserving high-precision optimizer state. Gradient checkpointing trades recomputation for fewer saved activations. Low-rank adaptation (LoRA) restricts updates to small adapters, reducing trainable parameters and optimizer state. ZeRO and Fully Sharded Data Parallel (FSDP) shard state across workers but introduce communication. Reading the binding term before selecting a technique keeps these tradeoffs explicit.

<NotationOverride scope="Practical Training Optimizations">
$N$ continues to mean model **parameter count** as in Scaling Laws (<XRef id="def-separable-scaling-law" />), overriding the standing block-count meaning. Here $L$ is depth in transformer blocks.
</NotationOverride>

## The training-memory equation

A forward-only inference run avoids gradients, optimizer state, and activations saved for backward, but it still needs transient activations, workspace, and often a KV cache in addition to the parameters. Training is dramatically heavier: the optimizer stores gradients and per-parameter moments, and the backward pass must re-read activations cached by the forward pass. The first move is to collect every training contribution into one equation, so that each later tool can be read as an edit to one of its terms.

<Theorem kind="definition" name="Training-memory equation" id="def-train-memory">
For a model of $N$ parameters trained by a stochastic optimizer, the total per-worker training memory decomposes as
$$
M_{\text{train}} \defeq \underbrace{N b_p}_{\text{params}} + \underbrace{N b_g}_{\text{gradients}} + \underbrace{N b_o}_{\text{optimizer state}} + \underbrace{A\, b_a}_{\text{activations}} + B_{\text{misc}},
$$
where $b_p, b_g, b_o, b_a$ are the bytes per parameter, gradient, optimizer-state slot, and activation element; $A$ is the activation count cached for backprop; and $B_{\text{misc}}$ collects fixed-size buffers (workspace, all-reduce staging, GEMM scratch).
</Theorem>

<MarginNote variant="formula" label="Adam state">Adam keeps two moments per parameter, so $b_o = 8$ bytes (two FP32 slots) regardless of the working precision $b_p$.</MarginNote>

The optimizer term is the one most people omit. We pin it down for the standard configuration — FP32 storage with Adam — which fixes the baseline every later tool is measured against.

<Theorem kind="proposition" name="FP32+Adam baseline" id="prop-fp32-adam">
For a model of $N$ parameters trained in FP32 with Adam (a first- and second-moment buffer per parameter), the parameter, gradient, and optimizer contributions to $M_{\text{train}}$ total
$$
M_{\text{train}}^{(\mathrm{P+G+O})} = 4N + 4N + 8N = 16N \quad \text{bytes}.
$$
</Theorem>

<Theorem kind="proof">
Three terms, each from the byte width of one quantity.

**Parameters.** FP32 stores each scalar in $4$ bytes, so $b_p = 4$ and $N b_p = 4N$.

**Gradients.** The backward pass writes one FP32 gradient per parameter, so $b_g = 4$ and $N b_g = 4N$.

**Optimizer.** Adam <Cite key="kingma2015adam" /> maintains a first-moment $m_t$ and a second-moment $v_t$ buffer per parameter, each in FP32, so $b_o = 8$ and $N b_o = 8N$.

Summing gives $16N$ bytes. $\qquad\blacksquare$
</Theorem>

The constant $16N$ is the unit of account for the rest of the chapter: a $70$B model in this baseline needs $16 \cdot 70 \times 10^9 = 1.12 \times 10^{12}$ bytes — $1.12$ TB — for parameters, gradients, and Adam state alone, before a single activation. That is roughly fourteen $80$ GB accelerators, which is why the bare baseline already forces the question every later term answers.

<MarginNote variant="pattern" label="16N rule">Multiply $N$ in billions by $16$ to get gigabytes for the master-copy block. "Does it fit on one GPU?" is then one mental operation.</MarginNote>

<Theorem kind="remark" name="Activations are invisible in the 16N count" id="rem-activations">
The $16N$ figure covers only the master-copy block; the activation term $A\,b_a$ is separate and frequently dominates. A coarse *floor* without checkpointing is $A\,b_a \approx L\, n\, d\, b_a$ per microbatch element, so for $L = 80$, $n = d = 8192$, $b_a = 2$ (bf16) the activations already reach $\approx 11$ GB per element before the per-block scratch factor (several-fold for the feed-forward and attention intermediates). At realistic batch sizes the activation term can exceed the $16N$ baseline — which is exactly what gradient checkpointing exists to attack.
</Theorem>

## Mixed precision and numerical range

The cheapest term to attack is the bytes per element. FP32 spends $4$ bytes per scalar; the $16$-bit formats spend $2$. But halving precision naively breaks training: a $16$-bit float with too few exponent bits cannot represent the small magnitudes that gradient-moment updates produce, and they underflow to zero. Mixed precision is the construction that takes the speed of $16$-bit arithmetic while keeping a high-precision copy where the numerics demand it.

<Theorem kind="definition" name="Mixed-precision training" id="def-mixed-precision">
**Mixed-precision training** <Cite key="micikevicius2018mixed" /> maintains an FP32 *master copy* of the parameters, casts a $2$-byte working copy (bf16 or fp16) for every forward and backward pass, and casts gradients back to FP32 before the optimizer update. The Adam first- and second-moment buffers remain FP32 throughout.
</Theorem>

<MarginNote variant="pattern" label="bf16 vs fp16">bf16 preserves FP32's exponent range. fp16 offers more mantissa precision but a narrower range, so bf16 is often the safer default when hardware and kernels support it; validate the choice for the workload.</MarginNote>

The qualitative reason bf16 is the safer default is a statement about dynamic range — which bits a format spends on magnitude versus precision.

<Theorem kind="remark" name="Why bf16 beats fp16 — a dynamic-range argument" id="rem-bf16">
The two 16-bit formats below store the same 16 bits but allocate them differently; FP32, shown for reference, has twice as many.

| Format | Sign | Exponent | Mantissa |
|---|---|---|---|
| FP32 | 1 | 8 | 23 |
| fp16 | 1 | 5 | 10 |
| bf16 | 1 | 8 | 7 |

Normal and subnormal ranges must be distinguished. fp16's smallest positive **normal** number is $2^{-14}\approx 6.10\times10^{-5}$, while its smallest positive **subnormal** is $2^{-24}\approx5.96\times10^{-8}$. Values below the normal floor have reduced precision, and accelerator kernels may flush subnormals to zero; small gradients can therefore vanish during fp16 backward arithmetic or fp16 gradient transport before they are converted to FP32. bf16 keeps FP32's $8$-bit exponent (hence essentially FP32's dynamic range) and pays for it in mantissa precision, so this transport problem is far less acute. fp16 training therefore uses *loss scaling* — multiply the loss by a power of two before backprop, then unscale after the gradients have safely reached FP32 — to protect the backward and transport path. It does **not** protect Adam's moment buffers: under <XRef id="def-mixed-precision" /> those moments and their update already remain FP32.
</Theorem>

The memory consequence is quantitative, and the surprise is that it is smaller than "$16$-bit, so half the memory" suggests: the FP32 master copy and FP32 Adam state stay put.

<Theorem kind="proposition" name="Mixed-precision memory" id="prop-mixed-precision">
With an FP32 master copy, a bf16/fp16 working copy, and FP32 Adam moments, the parameter+gradient+optimizer block of <XRef id="def-train-memory" /> becomes
$$
M_{\text{train}}^{(\mathrm{P+G+O})} = (4 + 2)N + 2N + 8N = 16N \quad \text{bytes}.
$$
Activations stored in bf16 contribute $b_a = 2$, halving the activation term from its FP32 value.
</Theorem>

<Theorem kind="proof">
Four contributions, from the four copies the scheme keeps live.

**Parameters.** The FP32 master copy contributes $4N$; the bf16 working copy that must coexist during the forward pass adds $2N$, for a $(4+2)N = 6N$ parameter term.

**Gradients.** Gradients are written in the $2$-byte working precision, so $b_g = 2$ and the gradient term is $2N$.

**Optimizer.** Adam's two moments stay FP32, so $b_o = 8$ and the optimizer term is $8N$ — unchanged from <XRef id="prop-fp32-adam" />.

Summing gives $6N + 2N + 8N = 16N$. Activations cast to bf16 use $b_a = 2$, half the FP32 width. $\qquad\blacksquare$
</Theorem>

So mixed precision leaves the master-copy block at $16N$ — the working copy's extra $2N$ is exactly offset by the gradients dropping from $4N$ to $2N$ — while halving activations. Its memory win is the activation term, not the master-copy block; its decisive win is elsewhere.

<Theorem kind="remark" name="The dominant gain is often throughput on supported hardware" id="rem-throughput">
On accelerators with native low-precision matrix units and suitable kernels, mixed precision often delivers a larger throughput gain than its modest master-copy memory change suggests. The speedup relative to FP32 depends on the device, datatype, kernel shapes, and whether another bottleneck binds; it is not a universal order-of-magnitude factor. In supported regimes, the wall-time saving can matter more than the memory accounting of <XRef id="prop-mixed-precision" />, but mixed precision remains an engineering choice to validate rather than a mathematical necessity.
</Theorem>

## Gradient checkpointing: trading activations for FLOPs

Mixed precision left the activation term $A\,b_a$ structurally intact (it only halved $b_a$). For a transformer of depth $L$, the forward pass writes one activation tensor per layer and the backward pass needs every one, so $A$ scales linearly in $L$. Gradient checkpointing keeps only a *subset* of these tensors and recomputes the rest by re-running the forward pass during backward — buying memory with arithmetic. The central fact is that one specific subset size is optimal, and it is provably $\sqrt{L}$.

<MarginNote variant="formula" label="sqrt(L) checkpointing">Store $\sqrt{L}$ of the $L$ layer activations: peak memory $\Theta(\sqrt{L})$ at the cost of one extra forward pass.</MarginNote>

<MarginNote variant="tip" label="Self-explanation">With $K$ checkpoints, one memory term is $K$ and the longest recomputed segment is about $L/K$. Which $K$ minimizes their sum?</MarginNote>

<Theorem kind="theorem" name="sqrt(L) checkpointing tradeoff" id="thm-checkpointing">
For positive integers $L$ and $K$ with $1\le K\le L$, split a sequential network of $L$ layers into $K$ contiguous segments whose lengths differ by at most one. Storing their boundary activations and recomputing within a segment gives peak activation memory
$$
\Theta\!\left(K+\left\lceil\frac{L}{K}\right\rceil\right),
$$
at the cost of one extra forward pass during the backward computation. The continuous relaxation $K+L/K$ is minimized at $K=\sqrt L$; choosing a legal neighboring integer $K\in\{\lfloor\sqrt L\rfloor,\lceil\sqrt L\rceil\}$ and using rounded segment boundaries yields peak memory $\Theta(\sqrt L)$. If an exactly equal-length presentation is desired, one may instead use a divisor of $L$ near $\sqrt L$ when such a divisor exists; divisibility is not required for the rounded construction. Any schedule storing $o(L)$ activations must recompute the missing ones, so the extra forward work is the price of the memory saving; the $O(\sqrt{L})$ construction achieving it is due to <Cite key="chen2016training" />.
</Theorem>

<Theorem kind="proof">
Two steps: the memory cost of a uniform schedule, then its minimization.

**Memory of a uniform schedule.** Put the boundaries at rounded multiples of $L/K$. The resulting $K$ segment lengths are $\lfloor L/K\rfloor$ or $\lceil L/K\rceil$; if $K\mid L$, they are exactly $L/K$. Store only the $K$ segment-boundary activations — this is the $K$ term. During the backward pass, process segments from last to first; to differentiate one segment, re-run its forward pass from its stored starting checkpoint, holding at most $\lceil L/K\rceil$ activations of *that one segment* live. Only one segment is materialized at a time, so peak activation memory is $\Theta(K+\lceil L/K\rceil)$.

**Minimization (AM–GM, then integer rounding).** The recompute pass touches each layer exactly once across the whole backward sweep, so the extra arithmetic is a single forward pass regardless of $K$. For the continuous relaxation $g(x)=x+L/x$ on $[1,L]$, AM–GM gives $x+L/x\ge2\sqrt L$, with equality at $x=\sqrt L$ (equivalently $g'(x)=1-L/x^2=0$). An admissible integer nearest $\sqrt L$ differs from it by at most $1$, and its ceiling term is at most $L/K+1$, so it attains $\Theta(\sqrt L)$ peak memory. When $L$ is a perfect square the exact continuous optimizer is itself legal and the unrounded value is $2\sqrt L$. $\qquad\blacksquare$
</Theorem>

The exchange the theorem prices is among the best available: a $\sim\!\sqrt{L}$-fold drop in activation memory (e.g. $L = 64 \to K = 8$: peak $K + L/K = 16$ live tensors instead of $64$, a factor of four) for a single extra forward pass. Since the backward pass is already roughly twice the cost of a forward, adding one more forward raises the step time by only about a third.

<Theorem kind="remark" name="Selective recomputation targets favorable tradeoffs" id="rem-selective">
Real training stacks need not use uniform $\sqrt{L}$ checkpointing. *Selective recomputation* retains activations whose regeneration cost is high relative to the bytes they occupy and recomputes intermediates whose storage cost is high relative to their regeneration cost. Depending on the model and kernels, recomputed tensors may include $O(n^2)$ attention probabilities or feed-forward intermediates. This policy can recover much of the memory benefit with less wall-time overhead than re-running every operation. <XRef id="thm-checkpointing" /> remains the baseline for a uniform layer schedule, but selective policies change tensor sizes and recomputation costs, so their improvement is not universally only a constant factor.
</Theorem>

## LoRA and trainable state

Mixed precision shrinks $b_p$ and $b_g$; checkpointing shrinks $A$. Under this chapter's FP32-Adam baseline, the optimizer term $N b_o$ remains the largest contributor in many configurations because $b_o=8$. Compressed optimizer states can change that premise, with their own numerical and implementation tradeoffs. LoRA attacks a different factor entirely — the *count* of trainable parameters. It freezes the pretrained weights and trains only a small additive update, and the right way to see the saving is that gradients and optimizer state exist only for what is trainable.

<Theorem kind="definition" name="LoRA adapter" id="def-lora">
For a pretrained weight matrix $\mathbf W_0 \in \R^{d\times d}$, a **LoRA adapter** <Cite key="hu2021lora" /> replaces $\mathbf W_0$ during fine-tuning by
$$
\mathbf W = \mathbf W_0 + \Delta\mathbf W, \qquad \Delta\mathbf W \defeq \mathbf B\mathbf A, \quad \mathbf A \in \R^{r\times d},\ \ \mathbf B \in \R^{d\times r},
$$
with rank $r \ll d$. Only $\mathbf A$ and $\mathbf B$ are trained; $\mathbf W_0$ is frozen. The initialization is asymmetric — $\mathbf A$ a small Gaussian and $\mathbf B = \mathbf 0$ — so $\Delta\mathbf W = \mathbf 0$ at step zero and the model begins identical to the pretrained checkpoint.
</Theorem>

<MarginNote variant="formula" label="Saving factor">For $r=16$ and $d=4096$, an adapted square matrix uses $1/128$ as many trainable parameters. The worked 70B example leaves about 0.08B trainable.</MarginNote>

The construction yields a clean count: trainable parameters, gradients, and optimizer state all shrink by the same factor.

<Theorem kind="proposition" name="LoRA parameter and optimizer-state savings" id="prop-lora-savings">
Suppose $K$ pretrained matrices $\mathbf W_0^{(i)} \in \R^{d\times d}$ are each replaced by a rank-$r$ adapter $\mathbf B^{(i)}\mathbf A^{(i)}$. Then:
1. Trainable parameters drop from $K d^2$ to $2Krd$, a reduction by the factor $\dfrac{2r}{d}$.
2. The gradient buffer and the Adam optimizer state shrink by the *same* factor $\dfrac{2r}{d}$, since the frozen $\mathbf W_0^{(i)}$ carry no gradient and no moments.
</Theorem>

<Theorem kind="proof">
Three counts, from definitions of the three quantities.

**Parameters.** Each adapter has $\mathbf A^{(i)} \in \R^{r\times d}$ ($rd$ entries) and $\mathbf B^{(i)} \in \R^{d\times r}$ ($dr$ entries), so $2rd$ trainable parameters per matrix. Over $K$ matrices that is $2Krd$, against $Kd^2$ for full fine-tuning; the ratio is $2r/d$.

**Gradients.** Backpropagation produces a gradient for every *trainable* parameter only. Since each $\mathbf W_0^{(i)}$ is frozen, gradients exist only for the $2Krd$ adapter parameters, so the gradient buffer scales by $2r/d$.

**Optimizer state.** Adam keeps two moment buffers per gradient. With gradients only for the $2Krd$ trainable parameters, the optimizer state likewise scales by $2r/d$. $\qquad\blacksquare$
</Theorem>

The arithmetic is dramatic at scale. Adapting the four attention projections in each of $L = 80$ layers of a $70$B model with $d = 8192$, $r = 16$ gives $K = 320$ matrices and $2Krd = 2 \cdot 320 \cdot 8192 \cdot 16 \approx 8.4 \times 10^{7}$ trainable parameters — about $0.12\%$ of the full count — and an FP32 Adam footprint of $8 \cdot 8.4 \times 10^7 \approx 0.67$ GB, against $8 \cdot 70 \times 10^9 = 560$ GB for full fine-tuning. The optimizer term, the stubbornest in the baseline, all but vanishes.

LoRA is the one tool here that explicitly restricts the trainable function class: each adapted matrix update has rank at most $r$. The memory count is exact, while whether that update class is rich enough is an empirical question.

<Theorem kind="remark" name="When LoRA's rank constraint matters" id="rem-lora-breaks">
The construction proves $\operatorname{rank}(\Delta\mathbf W)\le r$ for each adapted matrix; it does not identify a unique “true” full-fine-tuning update or a universal failure regime. If the useful change cannot be approximated at the selected ranks and module placements, the restriction can cost expressive power. Continued pretraining on a large distribution shift, adding a new modality or vocabulary, and choosing a very small rank are plausible stress cases, not guaranteed failures, and no numerical rank threshold alone decides adequacy. Rank, target modules, data, and optimization must therefore be validated together. For a static adapter at inference, one may merge $\mathbf W=\mathbf W_0+\mathbf B\mathbf A$ into one matrix, removing the adapter's extra matrix multiplications.
</Theorem>

## ZeRO and FSDP: sharding master-copy state

The previous tools shrink individual terms. ZeRO and FSDP attack a different axis: they partition selected parameter, gradient, and optimizer-state terms across $P$ data-parallel workers — at the cost of communication when partitioned state must be gathered or reduced. Activations and miscellaneous buffers are outside this stage definition. The family is organized into three stages by how much of the master-copy block it shards.

<Theorem kind="definition" name="ZeRO stages" id="def-zero">
The **ZeRO** family of memory optimizers <Cite key="rajbhandari2020zero" /> progressively partitions the master-copy block of <XRef id="def-train-memory" /> across $P$ data-parallel workers:
- **Stage 1** partitions the optimizer states: $N b_o \to N b_o / P$ per worker.
- **Stage 2** additionally partitions the gradients: $(N b_g + N b_o) \to (N b_g + N b_o)/P$.
- **Stage 3** additionally partitions the parameters: $(N b_p + N b_g + N b_o) \to (N b_p + N b_g + N b_o)/P$.

PyTorch's Fully Sharded Data Parallel (FSDP) is the production implementation of stage 3.
</Theorem>

The per-worker memory is immediate from the definition; the content is the *communication* each stage pays, which we compute from the collective each step performs.

<Theorem kind="proposition" name="ZeRO per-worker memory and communication" id="prop-zero-memory">
Let $M_P = N b_p$, $M_G = N b_g$, and $M_O = N b_o$ be the unsharded **resident-memory** contributions. Separately, let $b_w$ be the bytes per communicated parameter or gradient element on the wire. Under ring collectives and a ZeRO-3 schedule that reshards a layer's parameters after its forward use, the per-worker resident master-copy memory and payload volume per optimizer step are:

| Stage | Per-worker master copy | Communication / step |
|---|---|---|
| Distributed Data Parallel (DDP), no ZeRO | $M_P + M_G + M_O$ | $2\frac{P-1}{P}N b_w\approx2Nb_w$ |
| ZeRO-1 | $M_P + M_G + M_O/P$ | $2\frac{P-1}{P}N b_w\approx2Nb_w$ |
| ZeRO-2 | $M_P + M_G/P + M_O/P$ | $2\frac{P-1}{P}N b_w\approx2Nb_w$ |
| ZeRO-3 / FSDP | $(M_P + M_G + M_O)/P$ | $3\frac{P-1}{P}N b_w\approx3Nb_w$ |

The approximations are the large-$P$ limits. These are payload-byte counts per worker; they exclude collective latency, bucket padding, protocol overhead, and communication hidden by overlap with computation.
</Theorem>

<Theorem kind="proof">
The memory column is <XRef id="def-zero" /> applied term by term. The communication column comes from the collective each stage runs.

**DDP.** Every worker replicates the full master copy; only gradients are reduced, once per step. In a ring, each reduce-scatter and all-gather moves $(P-1)N b_w/P$ payload bytes per worker. Their composition therefore makes one all-reduce cost exactly $2(P-1)N b_w/P$ payload bytes per worker.

**ZeRO-1 and ZeRO-2.** Stage 1 partitions only optimizer state; whether implemented as the corresponding reduce-scatter, update, and all-gather sequence or an equivalent gradient synchronization, its two ring phases total $2(P-1)N b_w/P$. Stage 2 explicitly uses a reduce-scatter of gradients plus an all-gather of updated parameters. Each phase costs $(P-1)N b_w/P$, so the exact total is again $2(P-1)N b_w/P$.

**ZeRO-3 / FSDP.** Parameters are sharded. Under the stated reshard-after-forward schedule, each layer's full working weights are all-gathered once for its forward pass and again for its backward pass, and its gradients are reduce-scattered once. Each of these three ring phases costs $(P-1)N b_w/P$ after summing over layers, for an exact total $3(P-1)N b_w/P$. This is $1.5\times$ the DDP payload when both use the same wire width. $\qquad\blacksquare$
</Theorem>

So, under that schedule and a common $b_w$, ZeRO-3 buys full $1/P$ partitioning of the resident master-copy block for $1.5\times$ the payload of plain data parallelism. For a $70$B model in mixed precision the block is $(6 + 2 + 8)N = 16\cdot 70 \times 10^9 = 1120$ GB; on $P = 64$ workers its resident share is $17.5$ GB, below an $80$ GB accelerator's capacity. That figure alone does not show the full training job fits: saved activations, transient gathered layer parameters, communication buffers, and scratch space remain. DDP, by contrast, would replicate the $1120$ GB master-copy block on every worker and cannot fit it on one such device.

<Theorem kind="remark" name="Communication can become the binding constraint" id="rem-fsdp-bandwidth">
The exact $3(P-1)N b_w/P$ figure is payload bytes, not time; the time depends on interconnect bandwidth and latency, and ZeRO-3 stresses it hardest because the parameter all-gathers sit on the critical path of every layer. On a cluster with thin interconnect the all-gather latency for a layer can exceed that layer's compute time, leaving the accelerator idle. FSDP exposes a *wrapping granularity* knob — how many layers share one all-gather (per transformer block is the usual choice). Finer wrapping lowers peak memory but issues more, smaller collectives; coarser wrapping does the reverse. The right setting balances the two against the measured bandwidth, which is why ZeRO-3 is not a free lunch but a memory–bandwidth trade.
</Theorem>

## Putting it together

Each section above edits a specific part of <XRef id="def-train-memory" />: mixed precision changes storage widths (and halves $b_a$ in the stated scheme), checkpointing changes the saved-activation count $A$, LoRA changes the trainable-parameter count and therefore gradient and optimizer state, and ZeRO and FSDP partition the parameter, gradient, and optimizer block stage by stage. ZeRO does not generally divide $A b_a$ or $B_{\text{misc}}$ by $P$. The engineering question — which to reach for first — reduces to diagnosing the *binding* term before naming a tool.

<Theorem kind="remark" name="The bottleneck-first heuristic" id="rem-bottleneck">
Read the binding term off the equation, then pick the tool that edits it. A large master-copy block ("$70$B in FP32 $=1.12$ TB") points to mixed precision plus ZeRO. A large activation term ("long context $\Rightarrow$ large $A$") points to checkpointing. A small fine-tuning dataset with a frozen base often points to LoRA. In practice, mixed precision commonly comes first because bf16 often preserves convergence in supported regimes while improving throughput. ZeRO stages follow only as far as memory requires, since higher stages cost more communication. Checkpointing enters when activations bind, with selective and uniform policies compared by measurement. LoRA is commonly used for parameter-efficient fine-tuning; whether its rank-constrained updates suffice for continued pretraining or a large distribution shift is empirical rather than excluded by the method. Naming the bottleneck before the tool is the discipline.
</Theorem>

<SolutionBox exercise="14.1">

The master-copy block is $16N$ bytes: $4$ (FP32 params) $+ 4$ (FP32 grads) $+ 8$ (two FP32 Adam moments). For $N = 13\times10^9$, $16N = 208$ GB, which overflows an $80$ GB accelerator by $2.6\times$ — before any activations. $\qquad\blacksquare$

</SolutionBox>

**Exercise 14.2 (Compute).** A $7$B model is trained in mixed precision (bf16 working, FP32 master, FP32 Adam). Using <XRef id="prop-mixed-precision" />, give the parameter+gradient+optimizer footprint and the fraction of an $80$ GB accelerator it occupies. Contrast with the naive "$16$-bit, so half of $16N$" guess and say which term defeats it.

<SolutionBox exercise="14.2">

The mixed-precision block is $16N$: a $6N$ parameter term ($4N$ FP32 master $+ 2N$ bf16 working), a $2N$ bf16 gradient term, and an $8N$ FP32 Adam term. For $N = 7\times10^9$, $16N = 112$ GB $= 1.4\times$ an $80$ GB accelerator. The naive "16-bit halves $16N$" guess fails because the FP32 master copy persists alongside the working copy and the FP32 Adam state ($8N$, the largest term) is precision-pinned. $\qquad\blacksquare$

</SolutionBox>

**Exercise 14.3 (Prove).** For the continuous relaxation $g(x) = x + L/x$ of a uniform checkpointing schedule, prove $x = \sqrt{L}$ minimizes it and that the optimum is $2\sqrt{L}$; then explain how <XRef id="thm-checkpointing" /> chooses a legal integer $K$. For $L = 100$, state the number of stored tensors, the memory-reduction factor, and the wall-time penalty.

<SolutionBox exercise="14.3">

Minimize the continuous relaxation $g(x)=x+L/x$. By AM–GM, $x+L/x\ge2\sqrt L$, equality iff $x=\sqrt L$ (or $g'(x)=1-L/x^2=0$, $g''>0$). For general integer $L$, choose the better legal neighbor $K\in\{\lfloor\sqrt L\rfloor,\lceil\sqrt L\rceil\}$ and use segment length at most $\lceil L/K\rceil$ as in <XRef id="thm-checkpointing" />. For $L=100$, $K=10$ is already integral: store $10$ checkpoints, with peak memory $10+100/10=20$ live tensors instead of $100$ — a $5\times$ reduction — for one extra forward pass on backward ($\approx+1/3$ step time). $\qquad\blacksquare$

</SolutionBox>

**Exercise 14.4 (Derive).** A $13$B model with $d = 5120$ is fine-tuned with LoRA at rank $r = 8$, applied to the four attention projections in each of $L = 40$ layers. Using <XRef id="prop-lora-savings" />, count the trainable parameters and their fraction of the full $13$B, then distinguish the Adam-state reduction relative to all-parameter fine-tuning from the reduction relative to fully tuning only the selected matrices.

<SolutionBox exercise="14.4">

With $K = 4\cdot40 = 160$ adapted matrices, $r=8$, and $d=5120$, the exact trainable count is
$$
2Krd=2\cdot160\cdot8\cdot5120=13{,}107{,}200.
$$
Relative to all $13\times10^9$ model parameters this is $13{,}107{,}200/(13\times10^9)\approx0.1008\%\approx1/992$. The ratio $2r/d=1/320$ uses a different denominator: it compares the adapters with full fine-tuning of the **160 selected** $d\times d$ matrices, whose total is $160d^2$, not with all $13$B parameters. Adam stores a fixed number of moment bytes per trainable parameter, so its state inherits $1/320$ relative to full tuning of those selected matrices, or approximately $1/992$ relative to an all-parameter $13$B fine-tuning baseline. The frozen weights produce no gradients and hence no moments. $\qquad\blacksquare$

</SolutionBox>

**Exercise 14.5 (Compare).** A $70$B model is trained on $P = 64$ workers in mixed precision (bf16 working weights and gradients, FP32 master weights, FP32 Adam state). Using <XRef id="prop-zero-memory" />, give the per-worker master-copy memory under ZeRO-3 and under DDP, and state the dominant communication volume of each.

<SolutionBox exercise="14.5">

The unsharded $(6+2+8)N = 16N$ block for $N=70$ B is $1120$ GB. ZeRO-3/FSDP partitions all three resident terms by $P=64$: $17.5$ GB per worker — fits — while DDP replicates the full $1120$ GB per worker and is infeasible on an $80$ GB device. With bf16 payloads, $b_w=2$ bytes, DDP communicates
$$
2\frac{63}{64}(70\times10^9)(2)=275.625\times10^9\ \text{payload bytes per worker per step},
$$
whereas ZeRO-3 under the reshard-after-forward schedule communicates $3/2$ times that, $413.4375\times10^9$ bytes. The resident precisions determine the memory figures; the independent wire precision $b_w$ determines these communication figures. $\qquad\blacksquare$

</SolutionBox>
